# AetherMesh — Fixes to port to the original project

Found and fixed in the `E:\Projects\Mesh C` working copy on 2026-07-03. Both
trees (`E:\Projects\Mesh` original, `E:\Projects\Mesh C` copy) share identical
code, so **all of these bugs exist in the original too**. Listed in priority
order (most impactful first). File paths are relative to the repo root.

Each fix corresponds to a git commit in the Mesh C repo; the short hash is noted.

---

## Critical — radio / mesh (these broke messaging)

### 1. Unicast relays collide with ACKs — every ACK lost in a 3+ node mesh  (`5bf3ae6`)
`firmware/src/MeshRouter.cpp`, in `processIncomingPacket`, the "relay unicast packet for someone else" branch.
- **Problem:** when a node relays a unicast packet, it transmits the relay
  **immediately**. But the packet's actual recipient sends its ACK at the same
  instant (both are triggered by the same received packet), so the relay and the
  ACK collide at the sender and the ACK is destroyed. Result: 0% ACK success
  whenever a third node is in range. Two-node tests worked, which masked it as a
  "RAK not receiving" hardware problem for a long time.
- **Fix:** queue the relay with random jitter (`millis() + random(300,700)`)
  instead of sending instantly, so it doesn't step on the recipient's ACK. Also:
  when relaying an ACK, cancel any still-queued relay of the packet it
  acknowledges (the target already answered). And jitter Route Reply sends the
  same way (`random(100,400)`) — a broadcast RREQ triggers the target and every
  proxy simultaneously, same collision class. Broadcast rebroadcasts already had
  jitter, which is why group chat/telemetry never showed the bug.

### 2. RAK3401 1W firmware halts at boot  (`41ca8a6`)
`firmware/src/RadioManager.cpp`, constructor, `RAK3401_1W` default TX power.
- **Problem:** default TX power was 27 dBm. The SX1262 chip physically maxes at
  22 dBm (the board's "1W" is an external PA). RadioLib rejects 27,
  `radio->begin()` fails, node halts forever in the LED-blink error loop. The 1W
  build was dead on arrival.
- **Fix:** default `txPower = 22` for `RAK3401_1W`.

### 3. Changing settings can leave the receiver switched off  (`41ca8a6`)
`firmware/src/RadioManager.cpp`, `reinit()`.
- **Problem:** `reinit()` runs at every boot with values from NVS/app config. It
  returned early if any setter (`setFrequency`/`setBandwidth`/
  `setSpreadingFactor`/`setOutputPower`) failed — **before** the `startReceive()`
  at the end. So one bad/rejected value left the radio in standby with RX off. TX
  still worked (transmit re-arms itself), producing the classic "transmits but
  hears nothing" after a settings change.
- **Fix:** sanitize inputs first (clamp SF to 7–12, BW to 125/250/500, power to
  −9…22), then apply all setters even if some fail, and **always** call
  `startReceive()` at the end. Return a status flag but never skip re-arming RX.

### 4. Receiver can silently drop out of RX with no recovery  (`3e5a989`)
`firmware/src/RadioManager.cpp` (`loop()` + new state fields in `.h`).
- **Problem:** the code only re-armed RX after a TX completed or a CRC/header
  error. If the SX1262 fell out of RX mode any other way, the node stayed deaf
  forever.
- **Fix:** added an RX watchdog — if no receive activity of any kind (packet,
  CRC err, header err) for 60 s while idle, force `standby()` + `startReceive()`.
  Track `lastRxActivityTime` (updated on every RX event). Also added a 30 s health
  log line (mode, IRQ register, seconds since last RX, ambient RSSI, live
  SF/BW/freq/power) for diagnosis. Watchdog is a no-op on a healthy node.

### 5. Queued rebroadcasts dropped if the radio was busy  (`d7a74f8`)
`firmware/src/MeshRouter.cpp`, `loop()` (rebroadcast queue drain).
- **Problem:** a queued rebroadcast was marked inactive (consumed) **before** the
  send; if the radio was mid-transmit, `serializeAndSend` failed and the packet
  was silently lost.
- **Fix:** only deactivate the slot after a successful send; if busy, leave it
  queued and retry next loop; give up only after 5 s past its scheduled time.

## Critical — protocol / config

### 6. Remote-over-LoRa config silently rejected  (`8221b48`)
`proto/mesh.proto`, `proto/mesh.options`, `firmware/src/main.cpp`
(`onReceivedConfig`), app `AetherMeshRepository.sendRemoteConfig`.
- **Problem:** the app packed the admin password into `node_name` as
  `"name:password"`. nanopb caps `node_name` at 16 bytes, so any name+password
  over 15 chars failed `pb_decode` and the **whole** config packet was dropped.
  Also a security smell (password parsed from a name field).
- **Fix:** added a dedicated `config_password` field to `NodeConfig` in the proto
  (max_size 33 in mesh.options), regenerate nanopb, have the firmware verify that
  field directly, and have the app set it. Remove the `name:password`
  string-splitting.

### 7. Node rename silently fails  (`bfa62ae`)
`proto/mesh.options` (`node_name max_size`), app config form
(`ui/main/MainScreen.kt`).
- **Problem A:** app allows 16-char names but `node_name max_size:16` means 15
  chars + null — a 16-char name fails `pb_decode` and drops the whole config
  packet while the app shows "Config sent!". **Fix:** bump to `max_size:17` and
  regenerate nanopb (buffer becomes `char node_name[17]`; firmware's
  `nodeCustomName[17]` already matches).
- **Problem B:** the settings form's `LaunchedEffect` was keyed on the nodes
  list, so it repopulated every field from the DB on **every telemetry packet**,
  clobbering in-progress edits (and clearing the password box). **Fix:** populate
  the form once per connected node (guard with a "configLoadedForNode" state),
  and reset auth state only when the connected node actually changes.

## Firmware — robustness / correctness

### 8. Stack buffer overflow in OLED rendering  (`177c923`)
`firmware/src/main.cpp` (`updateDisplay`) and `firmware/src/BLEManager.cpp` (`init`).
- **Problem:** `sprintf(nodeStr, "Name: %s (0x%04X)", …)` into `char nodeStr[30]`
  overflows with a 16-char name (needs 32 bytes) — stack corruption on every
  display refresh. Several other unchecked `sprintf`s nearby.
- **Fix:** convert all to `snprintf` with `sizeof(buf)`; give the tight buffers a
  little headroom.

### 9. `saveSettings` calls silently reset power-save mode  (`177c923`)
`firmware/src/main.cpp`.
- **Problem:** two `saveSettings(...)` calls (initial BLE password set, and
  `onReceivedConfig`) omitted the trailing `powerSave` argument, so it defaulted
  to `false` — setting a password or pushing remote config turned power-save off.
- **Fix:** pass the current `powerSaveMode` through.

### 10. Duplicate chat messages on the phone  (`177c923`)
`firmware/src/main.cpp` (`onLoRaPacketReceived`) + `MeshRouter` needs a
`hasSeen()` accessor.
- **Problem:** every received LoRa packet was forwarded to the phone **before**
  the router's dedup check, so mesh rebroadcasts of the same packet arrived 2–3×
  and the app stored each as a separate message.
- **Fix:** expose the dedup cache (`bool hasSeen(sender,packetId)`), and skip the
  BLE forward if the packet is a duplicate or a reflection of our own
  (`sender_id == localNodeId`).

### 11. Screen timeout never fired  (`bb1348b`)
`firmware/src/main.cpp` (`updateDisplay`, `onLoRaPacketReceived`,
`onLoRaPacketTransmitted`, end of `setup`).
- **Problem:** every LoRa RX/TX updated `lastDisplayActivityTime`, and the mesh
  beacons every ~7–10 s, so 10/15/30/60 s timeouts never expired — only "Always
  Off" worked. Also power-save forced a 10 s screen-on window even when the user
  chose "Always Off".
- **Fix:** stop counting radio traffic as display activity (only the user button,
  new-message popup, and boot wake the screen); power-save now only *caps* the
  timeout at 10 s, never turns on a screen configured off.

### 12. Battery read spam  (`d7a74f8`)
`firmware/src/main.cpp` (`readBatteryLevel`, Heltec V4).
- **Problem:** called on every 1 Hz display refresh, each call toggles the
  divider GPIO, blocks 10 ms, and logs.
- **Fix:** cache the reading for 30 s.

### 13. Spurious "TX glitch" log on every send  (`d7a74f8`)
`firmware/src/RadioManager.cpp` (`sendPacket`, after the CAD scan).
- **Problem:** `scanChannel()` fires DIO1 (CAD_DONE), leaving a stale
  `operationDone` flag that `loop()` then misreports as a "LoRa TX glitch
  ignored" on every transmit.
- **Fix:** clear `operationDone = false` right after the CAD check (radio is in
  standby, so nothing genuine is pending).

## App features (optional to port, but useful for range work)

### 14. Two-direction signal reporting in ACKs  (`7befd28`)
Added `acked_rx_rssi`/`acked_rx_snr` to the `Ack` proto message; the ACKing node
stamps how it heard the ping, so the sender learns the outbound link quality (not
just the ACK's return signal). App logs both. Backward compatible (old firmware
sends 0 → treated as "not reported").

### 15. Range-test CSV export  (`9b4232a`, `7befd28`)
Restored the Export CSV button on the range test card; exports all logs as a real
file via FileProvider share sheet (was clipboard-only, which truncates past
~1 MB). CSV now has both link directions, and blank (not fake −140/−20) values on
timeouts. Requires a FileProvider entry in the manifest + `res/xml/file_paths.xml`.

### 16. Range-test GPS with speed/accuracy  (`0207949`)
The test now runs its own 1 s GPS listener for its duration (was only updating
while the Map tab was visible), logs the phone's own unfuzzed fix with speed
(m/s) and horizontal accuracy (m). DB migration added nullable columns.

### 17. DB downgrade crash guard  (`7f120da`)
`SQLiteOpenHelper.onDowngrade` now recreates the schema instead of throwing.
Relevant only if two builds with different DB versions share an applicationId
(see #19).

### 18. Radio profile presets  (`d72d6af`)
Fast/Balanced/Long range/Max range chips (SF9/10/11/12 at 125 kHz) on both the
local and remote config screens, so the whole mesh can be set to matching SF/BW
without slider mismatches. Also made the remote config dialog prefill from the
last settings pushed to each node instead of hardcoded defaults.

### 19. Distinct applicationId for the copy  (`225a52c`)
`app/app/build.gradle.kts` applicationId → `com.example.aethermesh.c`, label
"AetherMesh C". This is specific to keeping the copy installable alongside the
original — **do NOT port this to the original tree**; it should keep
`com.example.aethermesh`.

## Notes for whoever applies these

- After any `proto/mesh.proto` or `mesh.options` change, regenerate the
  firmware's nanopb files: `cd proto && python generate_proto.py` (writes
  `firmware/src/mesh.pb.{c,h}`, which are committed). The Android side
  regenerates automatically on build.
- Verify with `pio run -e heltec_v4 -e rak4631 -e rak3401_1w` and
  `./gradlew :app:assembleDebug`.
- Fixes **1–7** are the ones that make the mesh actually work; **8–13** are
  correctness/robustness; **14–18** are tooling/UX; **19** is copy-specific
  (don't port).
- All settings/proto changes are backward compatible with old nodes on the air,
  but **the whole mesh must run matching SF/BW/region** to communicate at all.

# Round 2 (same day, afternoon) — reliability & security batch

### 20. Automatic retransmits for un-ACKed messages  (`0e218b1`)
`firmware/src/MeshRouter.{h,cpp}`. Every want_ack packet got exactly one shot.
Now: 4-slot pending-ACK queue, up to 3 retransmits ~3s apart (same packet_id;
recipient dedup prevents double delivery). **Key enabler:** recipients now
re-ACK duplicates addressed to them (a duplicate means the sender is retrying
because the ACK was lost — previously duplicates were silently dropped so
retries could never succeed). `sendText` also no longer drops messages when no
route exists: it transmits anyway (1-hop recipients hear it) while requesting
the route. Limitation: same-id retries aren't re-relayed by intermediate nodes.

### 21. Real RAK battery reading  (`7850426`)
`firmware/src/main.cpp` `readBatteryLevel()`. RAK boards returned hardcoded
98%, masking dead batteries. Now reads VBAT on WB_A0 (1.5M/1M divider, 3.0V
internal ref, 12-bit), cached 30s.

### 22. Diagnostic beacon off by default  (`7850426`)
The raw "AMTE" beacon (7-10s, every node, forever) is behind
`-D ENABLE_RAW_BEACON` now. RX watchdog widened 60s→120s to match the quieter
mesh (telemetry is the heartbeat).

### 23. AES-GCM chat encryption  (`b23e204`)
App `AetherMeshRepository`. Was AES-256-ECB (deterministic, no integrity).
Now AES-256-GCM, random 12-byte IV prepended, 128-bit tag; decrypt falls back
to legacy ECB for old messages/peers. `encryptAES` returns null on failure and
`sendMessage` REFUSES to send (previously fell back to plaintext flagged as
encrypted!). Also `TextMessage.content max_size` 128→168 (max that fits LoRa's
255-byte frame) — encrypted payloads base64-expand and previously broke
pb_decode; encrypted input capped at 96 chars, plaintext stays 127.

### 24. BLE auth rate limiting  (`da789fe`)
`firmware/src/main.cpp`. 5 wrong passwords → 30s lockout (status queries still
answered). Cleartext-over-BLE remains; challenge-response deferred (needs
SHA256 on both MCU platforms + proto change).

### 25. DB reads off the main thread  (`43b165e` + import fix `0575927`)
App. `refreshData()` queries ran on the main thread on every packet. Now on a
dedicated single-thread dispatcher (order preserved). Startup purge moved too.

### 26. Location-type foreground service  (`43b165e`)
Manifest + `AetherMeshService`. Service declares
`connectedDevice|location` (guarded on fine-location grant, via
`ServiceCompat.startForeground`), so range-test GPS keeps updating while
backgrounded/screen off. Add `FOREGROUND_SERVICE_LOCATION` permission.

### 27. Secrets in Keystore-encrypted storage  (`628093a`)
App. Node passwords + ECDH private key moved from plaintext SharedPreferences
to `EncryptedSharedPreferences` (androidx.security:security-crypto
1.1.0-alpha06), with one-time migration from the plain file.

### 28. Map ping popups show both directions + speed  (`167352c`)
Range-test map markers now show ping-at-target RSSI and GPS speed (mph).

### 29. SF12 aborted mid-transmit; remote config hit the wrong node  (`7199b6e`)
Two field-reported bugs.
- **SF12 ("Max" profile) never worked:** `RadioManager` TX timeout was a fixed
  2000ms, but an SF12/125kHz packet is ~2.3s on air, so `loop()`'s hard timeout
  force-reset the radio mid-transmit every time. SF11 (~1.3s) squeaked under it,
  so "Long" worked. **Fix:** derive the timeout from `radio->getTimeOnAir(len)`
  + 50% margin per packet (stored in `txTimeoutMs`).
- **Remote config did nothing to the target:** `onBlePacketReceived` applied ANY
  config packet from the phone to the LOCAL connected node (and rebooted it)
  regardless of `recipient_id`, so remote config never went over the air.
  **Fix:** apply locally only when `recipient_id` is this node (or 0); else fall
  through to the LoRa send path so it routes to the target.
- **Related (retransmit interaction):** unicast ACKs are now sent on receipt,
  BEFORE processing — a config packet reboots inside its callback, so the old
  post-processing ACK never sent and (with retransmits) rebooted the target once
  per retry. ACK-first stops that.
